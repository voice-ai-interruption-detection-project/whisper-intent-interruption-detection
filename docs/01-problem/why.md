# 왜 이 문제인가?

## 배경: 음성 기반 상담 시스템의 확대

### 현실
- **Kakao i, Naver ClovaBot** 등 이미 상용화된 Voice Agent 존재
- **AWS Connect** 같은 클라우드 기반 솔루션도 음성 상담 지원
- **토스, 카뱅** 등의 채용 공고에서 "음성 기반 고객상담 AI" 관련 인재 모집 중
  → 실제로 구축/개선 중인 회사들이 늘어나는 중

---

## 현실의 문제: 어색한 대화

### 상황 1: 맞장구인데 멈춤
```
AI: "현재 배송 중이며 내일 오후 도착 예정입니다."
고객: "네, 알겠어요."  (동의)
AI: 🤐 (갑자기 멈춤)
고객: "???"
```

### 상황 2: 요청을 못 들음
```
AI: "배송 현황을 알려드리겠습니다."
고객: "아, 그게 아니라 환불받고 싶은데요."  (다른 요청)
AI: "현재 상품은 배송 중이고..."  (계속 배송 얘기)
고객: 😠
```

---

## 근본 원인: 사용자의 의도를 보지 않는다

### 현재 대부분의 시스템
1. 음성 감지 → 멈춤 (간단한 방식)
2. 또는 STT는 함 → 하지만 의도 판단 안 함

### 문제
- "네, 알겠어요" vs "환불받고 싶어요" 구분 불가
- 고객이 **뭐라고 말했는지**는 아는데, **의도가 뭔지**는 못 봄

### 따라서 필요한 것
```
음성이 있다
  ↓
뭐라고 말했는가 (텍스트)
  ↓
의도가 뭔가 (의도 분류)
  ↓
상황에 맞는 행동 (동의면 계속, 다른 요청이면 전환)
```

---

## 왜 지금 풀어야 하는가?

### 기술적 측면
- **STT**: OpenAI Whisper 등 오픈소스 모델도 충분히 사용 가능한 수준
- **의도/행동 판단**: LLM structured output, sentence embedding, rule selector 등 여러 후보를 조합 가능
- 하지만 **"고객 신호를 해석하고 AI의 다음 행동으로 연결"**하는 검증 구조는 별도로 설계해야 함

### 비즈니스 측면
- Voice Agent 도입 기업들이 증가
- 고객 만족도는 대화의 자연스러움에 영향
- 작은 개선(5~10%)도 운영 비용과 만족도에 큰 영향

### 연구적 측면
- 대화 시스템의 interruption handling은 잘 연구되지 않음
- 한국어 음성 상담 도메인 특화 데이터 부족
- 우리의 접근이 참고할 만한 사례가 될 수 있음
