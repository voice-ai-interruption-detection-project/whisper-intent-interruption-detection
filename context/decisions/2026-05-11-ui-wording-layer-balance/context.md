# Context

## 문제 인식

문서 정리 브랜치에서 세운 기준은 "제품 의도와 하네스 용어의 층위를 나눈다"였다. 그러나 이 기준을 UI에 옮기는 과정에서, 문서에서 설명하던 용어를 화면 텍스트로 너무 직접적으로 노출하는 문제가 생겼다.

대표적으로 아래 표현들이 화면에서 어색하게 보였다.

- `행동 판단 Workbench`
- `단일 판단 케이스(Scenario) 확인`
- `아직 실행 전`
- `텍스트 입력 판단`
- `87% 행동 정확도`
- `판단 케이스(Scenario) X Policy`

문제는 개별 번역어 하나라기보다, UI 표면별 목적이 다르다는 점을 충분히 반영하지 못한 것이다. 문서에서는 개념을 정의해야 하지만, 화면에서는 사용자가 지금 무엇을 보고 무엇을 누를지 바로 읽혀야 한다.

## 선택지

### 1. 문서 용어를 UI에 일관되게 병기한다

기각했다.

좌측 내비나 첫 안내처럼 개념 기준점을 제공하는 곳에는 병기가 도움이 된다. 하지만 큰 제목, 버튼, metric, table header까지 같은 방식으로 병기하면 화면이 용어집처럼 보이고, 실제 사용 흐름이 둔해진다.

예를 들어 `판단 케이스(Scenario)`는 좌측 필터에서 의미가 있지만, 모든 제목과 테이블에 반복되면 화면의 목적보다 용어 설명이 앞선다.

### 2. UI를 모두 쉬운 한국어로 바꾼다

기각했다.

`action_accuracy`, `expected_action`, `actual_action`, `policy`, `run_id` 같은 표현은 단순 화면 문구가 아니라 metric/schema/API/run artifact 계약과 이어져 있다. 이를 `행동 정확도`, `기준 행동`, `판단 행동`처럼 모두 풀어 쓰면 보기에는 친절해 보일 수 있지만, 결과 파일과 코드에서 쓰는 이름을 추적하기 어려워진다.

이 프로젝트 UI는 일반 고객용 제품 화면이 아니라 판단 구조를 검증하는 작업 화면이므로, 일부 기술 표면은 계약 이름을 보존하는 편이 더 명확하다.

### 3. UI surface별로 병기, 자연어, 계약 이름을 나눠 쓴다

채택했다.

기준은 정답 단어 목록이 아니라 표면별 역할이다.

- 좌측 내비, 필터, 첫 진입 지점: 사용자가 개념 기준을 잡아야 하므로 선택적 병기를 허용한다.
- 큰 제목: 화면 목적을 짧고 자연스럽게 드러낸다.
- 버튼: 개념 설명보다 사용자가 실행하는 동작을 말한다.
- metric/table/schema/API/run artifact 표면: 계약 이름을 보존하는 편이 더 명확한지 먼저 본다.
- 한국어화: 모든 용어를 번역하는 규칙이 아니라, 이해를 돕기 위한 수단으로 쓴다.

## 실제 반영

이번 판단은 `6351d9e refactor(ui): UI wording 층위 정리`에 반영됐다.

주요 방향:

- 상단 제품명은 `Whisper Intent Workbench`로 정리했다.
- 큰 제목은 `단일 케이스 확인`처럼 자연스러운 화면 목적을 우선했다.
- 좌측에는 `판단 케이스(Scenario) 필터`, `판단 케이스(Scenario)`를 남겨 사용자가 개념을 잡을 수 있게 했다.
- 버튼은 `Text Replay 실행`, `Audio File Test 실행`, `선택한 policy 실행`, `모든 policy 실행`처럼 실행 동작으로 정리했다.
- metric/table header는 `action_accuracy`, `event_type`, `expected_action`, `actual_action`, `primary_failure`처럼 계약 이름을 보존했다.
- `Focus policy`처럼 선택 상태 설명으로는 어색한 문구는 `Policy 선택`으로 바꿨다.

## 판단이 바뀐 지점

처음에는 사용자의 "문서 원칙을 UI에도 반영하자"는 요청을 "문서에서 정리한 한국어 설명과 병기를 화면에도 넓게 적용한다"로 해석했다.

하지만 사용자가 스크린샷으로 지적한 문제는, 문서 용어를 화면에 많이 넣는 것이 오히려 화면의 품질을 낮춘다는 점이었다. 이 피드백 이후 기준은 "문서와 UI가 같은 단어를 쓰는가"에서 "각 표면이 자기 역할에 맞는 말을 하는가"로 바뀌었다.

특히 이번 사례는 하네스 문서 기준만의 문제가 아니었다. 문서에는 층위를 나눠야 한다는 실마리가 있었지만, AI 작업자가 그 기준을 UI 표면에 기계적으로 적용하며 생긴 문제에 더 가까웠다.
