# Raw Material

이 파일은 2026-05-11 UI wording 조정 대화에서 나중에 다시 봐야 할 피드백과 작업 판단만 발췌한다. 긴 대화 전문과 로컬 절대경로는 저장하지 않는다.

## 사용자 피드백 발췌

사용자는 문서 기준을 UI에 반영하자는 요청 이후, 첫 적용 결과가 너무 기계적으로 보인다고 피드백했다.

문제가 된 화면 예시:

- 상단 타이틀의 `행동 판단 Workbench`
- 큰 제목의 `단일 판단 케이스(Scenario) 확인`
- 결과 카드의 `아직 실행 전`
- 버튼의 `텍스트 입력 판단`
- run 카드의 `행동 정확도`
- table/section header의 과한 `판단 케이스(SCENARIO)` 병기

사용자 표현의 핵심:

```text
이런 ... 행동판단 Work Bench 는 너무 짜치는데
그리고 아직 실행전은 뭐지? 이것도 어색해
...
5번째는 또 너무 행동정확도 이런식으로 표기햇네?
이건 보통 쓰는 기술적용어인데 이건 왜 그런거지?
...
일단 무조건적인 병기도 좀 있어 보이네
```

이후 사용자는 문제가 어디서 왔는지 물었다.

```text
혹시 이 패턴들은 내가 지금 이 대화세션에서 잘못 오해하게 지시해서 그런거야?
아니면 내 하네스 문서나 내용에 기준때문에 그런거야?
```

답변의 요지는 다음이었다.

- 사용자 지시가 틀렸다기보다, "문서 원칙을 UI에도 반영"이라는 요청을 AI가 너무 문자 그대로 적용했다.
- 하네스 문서에는 제품 의도와 용어 층위를 나눠야 한다는 기준이 있었지만, UI 표면별 자연스러움까지 명시되어 있지는 않았다.
- 따라서 원인은 문서 기준 자체보다, 문서 용어를 UI에 옮길 때의 적용 판단 부족에 가깝다.

## 기준 서버 비교

사용자는 브랜치 작업 전 기준 코드와 현재 브랜치를 로컬 서버 두 개로 비교하고 싶다고 했다.

- 현재 브랜치 서버: `http://127.0.0.1:8000`
- 작업 전 기준 서버: `http://127.0.0.1:8001`

비교 후 사용자는 현재 기준이 더 낫다고 판단했다. 다만 좌측에는 `판단 케이스`와 `Scenario`를 함께 병기하고, `Focus policy`는 어색하므로 `Policy 선택`처럼 정리하자고 했다.

## 최종 기준으로 정리한 내용

사용자가 제안한 핵심 문장:

```text
UI wording을 볼 때는 문서 용어를 그대로 많이 노출하는 것이 정합성이 아니다.
좌측 내비/필터/첫 안내처럼 개념 기준점이 되는 곳은 병기를 허용하되,
큰 제목, 버튼, metric, table header는 사용 맥락을 우선한다.
특히 action_accuracy, expected_action, actual_action, policy, run_id 같은
schema/metric/API 표면은 억지 번역하지 않는다.
```

이 문장을 너무 정답 사례처럼 고정하지 않기 위해, 최종 rule/agent에는 아래처럼 더 얇은 참고 관점으로 반영했다.

- 병기는 사용자가 개념을 처음 잡는 데 도움이 되는지 먼저 본다.
- 제목은 화면의 목적을 짧게 드러내는지 본다.
- 버튼은 개념 설명보다 사용자가 실행하는 동작으로 읽히는지 본다.
- metric/schema/API 표면은 계약 이름을 보존하는 편이 더 명확할 수 있다.
- 한국어화는 자연스러운 이해를 돕기 위한 수단이지, 모든 용어를 번역해야 한다는 규칙이 아니다.

## 반영 로그

- UI wording과 정적 테스트를 조정했다.
- 코드 주석/docstring의 하네스 용어 표현을 정리했다.
- `context-language-balancer` agent에 UI wording 점검 관점을 추가했다.
- `.claude/rules/coding.md`에도 같은 내용을 엄격한 규칙이 아니라 참고할 관점으로 추가했다.
